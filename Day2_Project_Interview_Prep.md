### Java vs JavaScript

- [ ] **"How are Java and JavaScript different?" / "Java và JavaScript khác nhau như thế nào?"**

> *Like "Car" and "Carpet" — they only share a name, they are completely different!*
>
> *Giống như "Car" và "Carpet" — chỉ giống tên, hoàn toàn khác nhau!*

| | Java | JavaScript |
|---|---|---|
| Data types / Kiểu dữ liệu | Static (declare type first) / Static (khai báo kiểu trước) | Dynamic (auto-detect type) / Dynamic (tự động nhận kiểu) |
| Runs on / Chạy trên | JVM (Java Virtual Machine) / JVM (máy ảo Java) | Browser / Node.js |
| OOP | Class-based (traditional) / Class-based (chính thống) | Prototype-based |
| Threading / Đa luồng | Multi-threaded | Single-threaded (event loop) |
| Used for / Dùng cho | Backend, Android, Enterprise | Frontend web, Full-stack |
| Compilation / Biên dịch | Compiles to bytecode first / Compile ra bytecode trước | Interpreted directly / Interpret trực tiếp |

```java
// Java — must declare type / phải khai báo kiểu
int tuoi = 22;           // MUST say "int" / PHẢI nói là "int"
String ten = "Minh";     // MUST say "String" / PHẢI nói là "String"
```

```javascript
// JavaScript — no type declaration needed / không cần khai báo kiểu
let tuoi = 22;           // auto-understood as number / tự hiểu là số
let ten = "Minh";        // auto-understood as string / tự hiểu là chuỗi
tuoi = "hai mươi hai";   // can change to string too! (dynamic) / đổi thành chuỗi cũng được! (dynamic)
```

### REST API & HTTP

- [ ] **"What is a REST API?" / "REST API là gì?"**

> *REST API is like a restaurant menu. The customer (client) orders food (request), the kitchen (server) prepares and serves the dish (response). Each dish has its own code (URL).*
>
> *REST API giống như thực đơn nhà hàng. Khách hàng (client) gọi món (request), bếp (server) làm và trả món (response). Mỗi món có mã số riêng (URL).*

**HTTP Methods — "Types of requests" / "Các loại yêu cầu":**

| Method | Like... / Giống như... | Purpose / Mục đích | Example / Ví dụ |
|---|---|---|---|
| **GET** | View menu / Xem thực đơn | Read data / Đọc dữ liệu | View product list / Xem danh sách sản phẩm |
| **POST** | Order a new dish / Gọi món mới | Create new data / Tạo dữ liệu mới | Create new account / Tạo tài khoản mới |
| **PUT** | Replace entire dish / Đổi toàn bộ món | Update entirely / Cập nhật toàn bộ | Update all user info / Sửa toàn bộ thông tin user |
| **PATCH** | Add seasoning / Thêm gia vị | Update partially / Cập nhật 1 phần | Only change user name / Chỉ đổi tên user |
| **DELETE** | Cancel order / Huỷ món | Delete data / Xoá dữ liệu | Delete account / Xoá tài khoản |

**Status Codes — "The restaurant replies" / "Nhà hàng trả lời":**

| Code | Like... / Giống như... | Meaning / Ý nghĩa |
|---|---|---|
| **200** | "Here's your food!" / "Đây món ăn của bạn!" | OK — success / OK — thành công |
| **201** | "Your new order has been noted!" / "Món mới đã được ghi!" | Created — created successfully / Created — tạo thành công |
| **400** | "I don't understand your order" / "Tôi không hiểu bạn gọi gì" | Bad Request — invalid request / Bad Request — yêu cầu sai |
| **401** | "You haven't logged in / reserved a table" / "Bạn chưa đăng nhập/đặt bàn" | Unauthorized — not authenticated / Unauthorized — chưa xác thực |
| **403** | "You're not allowed in VIP" / "Bạn không được phép vào VIP" | Forbidden — no permission / Forbidden — không có quyền |
| **404** | "This dish is not on the menu" / "Món này không có trong thực đơn" | Not Found / Not Found — không tìm thấy |
| **500** | "The kitchen is on fire!" / "Bếp bị cháy rồi!" | Internal Server Error / Internal Server Error — lỗi server |

```
GET    /api/users          → Get list of users / Lấy danh sách user
GET    /api/users/123      → Get user with id = 123 / Lấy user có id = 123
POST   /api/users          → Create new user / Tạo user mới
PUT    /api/users/123      → Update entire user 123 / Cập nhật toàn bộ user 123
DELETE /api/users/123      → Delete user 123 / Xoá user 123
```

### Git Basics — Git Cơ Bản

- [ ] **"How do you use Git?" / "Bạn sử dụng Git như thế nào?"**

> *Git is like a "super Undo/Save button" for code. You can go back to any point in time, and multiple people can work together without overwriting each other.*
>
> *Git giống như "nút Undo/Save siêu cấp" cho code. Bạn có thể quay lại bất kỳ thời điểm nào, và nhiều người có thể cùng làm mà không đè lên nhau.*

**Imagine this:** Think of writing a group thesis:

**Hình dung như này:** Nghĩ về viết luận văn nhóm:
- `git clone` = **Copy the thesis** from Google Drive to your machine / **Copy luận văn** từ Google Drive về máy
- `git branch` = **Create your own draft** to write your part / **Tạo bản nháp riêng** để viết phần của mình
- `git add` = **Mark** the changes you want to keep / **Đánh dấu** những thay đổi muốn giữ
- `git commit` = **Save** with a note like "finished section 2" / **Lưu lại** với ghi chú "đã viết xong phần 2"
- `git push` = **Push up** to Google Drive for everyone to see / **Đẩy lên** Google Drive cho mọi người thấy
- `git pull` = **Download** the latest version from Google Drive / **Tải về** bản mới nhất từ Google Drive
- `git merge` = **Merge** your part into the main version / **Gộp** phần của mình vào bản chính

```bash
# Start working / Bắt đầu làm việc
git clone https://github.com/team/project.git   # copy repo về
cd project

# Create your own branch for a feature / Tạo nhánh riêng để làm feature
git checkout -b feature/login                     # tạo nhánh mới

# Done coding, save your work / Viết code xong, lưu lại
git add LoginService.java                         # đánh dấu file thay đổi
git commit -m "Add login feature with JWT token"  # lưu với ghi chú

# Push to remote / Đẩy lên remote
git push origin feature/login                     # đẩy lên GitHub

# When you want to merge into main / Khi muốn gộp vào main
git checkout main
git pull origin main                              # lấy bản mới nhất
git merge feature/login                           # gộp code vào
```

**Branching Strategy (teamwork / làm việc nhóm):**
- `main` — main code, always runnable / code chính, luôn chạy được
- `develop` — code under development / code đang phát triển
- `feature/xxx` — branch for new features / nhánh làm tính năng mới
- **Pull Request (PR)** — request to merge code, others review before merging / xin phép gộp code, người khác review trước khi merge

### MVC Pattern

- [ ] **"What is MVC?" / "MVC là gì?"**

> *MVC divides an application into 3 separate parts, each responsible for one thing.*
>
> *MVC chia ứng dụng thành 3 phần riêng biệt, mỗi phần lo 1 việc.*

**Imagine this:** Think of a **restaurant**:

**Hình dung như này:** Nghĩ về **nhà hàng**:
- **Model (Kitchen / Bếp)** = Where food is cooked, ingredients are processed. In code: handles data, business logic, connects to database. / Nơi nấu ăn, xử lý nguyên liệu. Trong code: xử lý dữ liệu, business logic, liên hệ database.
- **View (Dining table / Bàn ăn)** = Where the customer sees the beautifully presented dish. In code: user interface (HTML, UI). / Nơi khách thấy món ăn đẹp mắt. Trong code: giao diện người dùng (HTML, UI).
- **Controller (Waiter / Phục vụ)** = The person who takes the order from the customer, passes it to the kitchen, and brings the dish out. In code: receives request, calls logic, returns response. / Người nhận order từ khách, chuyển cho bếp, mang món ra. Trong code: nhận request, gọi logic, trả response.

```
User (Customer / Khách hàng)
    ↓  click/input / click/nhập liệu
Controller (Waiter / Phục vụ) — receives request / nhận yêu cầu
    ↓  calls processing / gọi xử lý
Model (Kitchen / Bếp) — processes data, computes / xử lý dữ liệu, tính toán
    ↓  returns result / trả kết quả
View (Dining table / Bàn ăn) — displays to user / hiển thị cho người dùng
```

```java
// Model — handles data / xử lý dữ liệu
class SinhVien {
    private String ten;
    private double diem;
    // getters, setters, business logic...
}

// Controller — receives request, coordinates / nhận request, điều phối
@RestController
class SinhVienController {
    @GetMapping("/sinhvien/{id}")
    public SinhVien layThongTin(@PathVariable int id) {
        return sinhVienService.timTheoId(id);
    }
}

// View — display (HTML/JSON returned to client) / hiển thị (HTML/JSON trả về cho client)
```

### Dependency Injection — Tiêm Phụ Thuộc

- [ ] **"What is Dependency Injection?" / "Dependency Injection là gì?"**

> *Instead of going out to buy ingredients yourself (creating dependencies), you get them delivered to your door (injecting dependencies).*
>
> *Thay vì tự đi mua nguyên liệu (tạo dependency), bạn được giao hàng tận nơi (inject dependency).*

**Imagine this: / Hình dung như này:**

**WITHOUT DI:** You cook by yourself — you have to go to the market to buy meat, vegetables, and spices. You depend on a specific market — hard to change.

**KHÔNG có DI:** Bạn tự nấu ăn — phải tự đi chợ mua thịt, mua rau, mua gia vị. Bạn phụ thuộc vào chợ Cũ → khó thay đổi.

**WITH DI:** You order delivery — someone delivers ingredients to your door. Want to switch ingredients? Just change the order, no need to change how you cook.

**CÓ DI:** Bạn gọi ship — người ta giao nguyên liệu tận nơi. Muốn đổi nguyên liệu? Chỉ cần đổi đơn hàng, không cần thay đổi cách nấu.

```java
// ===== WITHOUT DI — "go shopping yourself" / KHÔNG có DI — "tự đi chợ" =====
class DichVuDonHang {
    // Create PaymentService yourself — hard dependency (hard to test, hard to change)
    // Tự tạo PaymentService — phụ thuộc cứng (hard to test, hard to change)
    private DichVuThanhToan thanhToan = new DichVuThanhToan();

    void xuLyDon() {
        thanhToan.thanhToan();
    }
}
// Problem: want to switch to VNPay? Must EDIT code inside DichVuDonHang!
// Vấn đề: muốn đổi sang VNPay? Phải SỬA code trong DichVuDonHang!

// ===== WITH DI — "delivered to your door" / CÓ DI — "được giao tận nơi" =====
class DichVuDonHang {
    private DichVuThanhToan thanhToan;

    // ThanhToan is PASSED IN from outside (injected)
    // ThanhToan được TRUYỀN VÀO từ bên ngoài (inject)
    DichVuDonHang(DichVuThanhToan thanhToan) {
        this.thanhToan = thanhToan;
    }
}
// Want to switch to VNPay? Just pass a different object, NO need to edit DichVuDonHang!
// Muốn đổi sang VNPay? Chỉ cần truyền object khác, KHÔNG sửa DichVuDonHang!

// ===== Spring Framework does DI automatically / Spring Framework làm DI tự động =====
@Service
class DichVuDonHang {
    @Autowired  // Spring automatically "injects" DichVuThanhToan / Spring tự động "tiêm" DichVuThanhToan vào
    private DichVuThanhToan thanhToan;
}
```

**Benefits:** Easy to test (pass mock objects), easy to change, cleaner code.

**Lợi ích:** Dễ test (truyền mock object), dễ thay đổi, code sạch hơn.

### SQL Basics — SQL Cơ Bản

- [ ] **Joins — Joining tables / Nối bảng**

> *A Join is like merging 2 Excel spreadsheets based on a common column.*
>
> *Join giống như ghép 2 bảng Excel lại với nhau dựa trên cột chung.*

**Imagine this:** You have 2 tables:

**Hình dung như này:** Bạn có 2 bảng:
- Table `customers` / Bảng `khách_hàng`: id, name / tên
- Table `orders` / Bảng `đơn_hàng`: id, customer_id / khách_hàng_id, product / sản_phẩm

| INNER JOIN | LEFT JOIN | RIGHT JOIN |
|---|---|---|
| Only get customers who HAVE purchased / Chỉ lấy khách hàng ĐÃ mua hàng | Get ALL customers, even those who haven't purchased / Lấy TẤT CẢ khách hàng, kể cả chưa mua | Get ALL orders, even those without a customer / Lấy TẤT CẢ đơn hàng, kể cả đơn không có khách |
| Intersection of 2 tables / Giao của 2 bảng | Entire left table + intersection / Toàn bộ bảng trái + phần giao | Entire right table + intersection / Toàn bộ bảng phải + phần giao |

```sql
-- INNER JOIN: Only get customers WITH orders / Chỉ lấy khách hàng CÓ đơn hàng
SELECT kh.ten, dh.san_pham
FROM khach_hang kh
INNER JOIN don_hang dh ON kh.id = dh.khach_hang_id;
-- Result: only customers who have purchased / Kết quả: chỉ những khách hàng đã từng mua

-- LEFT JOIN: Get ALL customers, whether they purchased or not / Lấy TẤT CẢ khách hàng, có mua hàng hay không
SELECT kh.ten, dh.san_pham
FROM khach_hang kh
LEFT JOIN don_hang dh ON kh.id = dh.khach_hang_id;
-- Result: all customers, those who haven't purchased have san_pham = NULL / Kết quả: tất cả khách hàng, ai chưa mua thì san_pham = NULL
```

- [ ] **Indexing — Creating indexes / Đánh chỉ mục**

> *An index is like a book's table of contents — instead of reading from page 1 to page 500, you look up the table of contents and find it immediately.*
>
> *Index giống như mục lục sách — thay vì đọc từ trang 1 đến trang 500, bạn tra mục lục là tìm thấy ngay.*

- **Pros / Lợi:** SELECT queries are much faster / Truy vấn SELECT nhanh hơn nhiều
- **Cons / Hại:** INSERT/UPDATE is slower (must update the index) / INSERT/UPDATE chậm hơn (phải cập nhật mục lục)
- **Use when / Dùng khi:** Columns frequently used in WHERE, JOIN, ORDER BY / Cột thường dùng trong WHERE, JOIN, ORDER BY

- [ ] **Normalization — Chuẩn hoá**

> *Reduce data duplication — like not writing the company address on every employee row, but instead separating it into a company table.*
>
> *Giảm trùng lặp dữ liệu, giống như không viết địa chỉ công ty ở mỗi dòng nhân viên, mà tách riêng thành bảng công_ty.*

- **1NF:** Each cell contains only 1 value (no lists in a single cell) / Mỗi ô chỉ chứa 1 giá trị (không có danh sách trong 1 ô)
- **2NF:** 1NF + every column depends on the ENTIRE primary key / 1NF + mỗi cột phụ thuộc vào TOÀN BỘ khoá chính
- **3NF:** 2NF + no transitive dependencies (if A -> B -> C, then separate B) / 2NF + không có phụ thuộc gián tiếp (A → B → C thì tách B ra)

  - "Basically, it works like this: ___"
  - "Think of it as ___ (analogy)"
  - "The main difference between A and B is ___"
  - "I chose ___ because ___"

- [ ] **When you don't know: / Khi không biết:**
  - "I'm not entirely sure about that, but I think ___"
  - "I haven't worked with that directly, but I would approach it by ___"
  - "That's a great question. I don't know the exact answer, but I would look into ___"

- [ ] **When talking about a project: / Khi nói về project:**
  - "I was responsible for ___"
  - "The main challenge was ___"
